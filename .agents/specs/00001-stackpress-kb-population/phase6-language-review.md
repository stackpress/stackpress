# Phase 6 Language Review

## Purpose

Separate source-backed technical language from public positioning that requires
founder acceptance. This avoids blocking technical KB promotion on marketing
choices while preventing inferred slogans from becoming project truth.

## Recommended Technical Vocabulary

| Phrase | Use |
| --- | --- |
| server-capability composition framework | canonical internal architecture model |
| server-side-first capability authority | explains server-first without denying browser interactivity |
| executable generated client package | explains generated output returning to runtime |
| package-owned generation | links transform ownership to runtime responsibility |
| lifecycle-owned registration | explains config/listen/route/idea package boundaries |
| event capability bus | explains shared internal invocation across surfaces |
| executable operational policy | explains TypeScript configuration |
| host-routed React | explains Ingest routes plus Reactus rendering |
| serialized server snapshot | explains SSR-to-hydration props |
| schema-history reconciliation | explains revisions without migration-ledger overclaim |
| explicit adapter portability | explains host/driver portability boundaries |
| contract-chain compatibility | explains generated and cross-package drift |
| ownership-first contribution | explains contributor routing |

These phrases are accepted for technical explanation. They should not be used as
public taglines unless the founder accepts them for that purpose.

## Safe Public Facts

- Stackpress is server-side-first, event-driven, pluggable, and model-informed.
- It coordinates generated server/data/UI capabilities from Idea declarations.
- Applications can expose shared server capabilities through multiple interfaces.
- It combines focused open-source libraries rather than hiding all behavior in
  one core package.
- Generated code remains inspectable and is consumed by the runtime.
- Configuration and custom plugins remain normal parts of application design.

Public copy must preserve the limits attached to each statement.

## Founder Decisions Needed

### Positioning Category

Current source calls Stackpress a content management framework, while the
implementation supports broader application composition. Decide whether public
positioning remains CMS-first, becomes application-framework-first, or uses a
qualified combination.

### Primary Promise

Choose which benefit leads public messaging:

- coordinated capabilities across interfaces;
- model-informed generation with inspectable output;
- server-side-first extensibility;
- package-oriented application composition.

The code supports all four, but the source does not choose their marketing order.

### Public Terminology

Decide whether any technical phrases become branded language. Until then, use
them as explanations rather than capitalized product concepts.

## Rejected Or Restricted Phrases

| Phrase | Disposition |
| --- | --- |
| one model, many interfaces | incomplete; config and plugins are also causal |
| product contract | too broad for Idea alone |
| operations console | mischaracterizes configuration |
| schema generates the whole application | false overstatement |
| automatic API/admin/AI exposure | exposure is generated or configured by surface |
| write once, run anywhere | unsupported blanket portability claim |
| zero code | custom config, plugins, pages, and operations remain intentional |

## Phase 6 Handling

Context can promote technical vocabulary with explicit labels and record the
positioning question as unresolved. Marketing artifact tests should use safe
facts and show placeholders for category/promise decisions rather than inventing
a final tagline.
