# Initial Concept Dispositions

## Purpose

Resolve the first-pass C-001 through C-012 hypotheses after all library and
Stackpress source research. Dispositions guide deeper investigation; they do not
promote these phrases into context.

| Concept | Disposition | Research conclusion |
| --- | --- | --- |
| C-001 Model-to-surfaces framework | Revise | Strong model-to-capabilities-and-surfaces loop exists, but config and events are as important as the model. |
| C-002 `.idea` as product contract | Split and rename | Idea is a semantic coordination/schema transport format; Stackpress makes it a rich cross-package model contract. “Product contract” overstates the parser alone. |
| C-003 Event names as operational API | Confirm and sharpen | Event names are Stackpress's capability bus across commands, domain actions, pages, API, MCP, and lifecycle. Governance remains open. |
| C-004 Lifecycle boundaries define package ownership | Confirm | Package runtime and transforms consistently align with `config`, `listen`, `route`, and `idea` responsibilities. |
| C-005 Generation is additive and plugin-owned | Confirm | Idea delegates artifact semantics; Stackpress packages discover and append their own transforms. |
| C-006 Generated client as shared runtime contract | Confirm and strengthen | It is an executable runtime registry/package, not merely a type/code bridge. |
| C-007 Configuration as operations console | Rename | Source supports “operational policy composition.” “Operations console” implies a UI and should be rejected as current language. |
| C-008 SQL as lifecycle, not only query syntax | Split | Inquire owns visible SQL intent; Stackpress SQL adds generated domain and database lifecycle. The distinction is essential. |
| C-009 Server-aware React infrastructure | Revise | Reactus provides host-routed SSR/hydration infrastructure; Stackpress binds it to Ingest routes and server props. |
| C-010 Layered access surfaces over one model | Revise | Surfaces adapt shared model, config, and event capabilities. “One model” alone is incomplete. |
| C-011 Bootstrap-first and agent-friendly distribution | Confirm | Root CLI is dependency-free for scaffold/skills and ships agent workflows as product assets. |
| C-012 Focused companion libraries as foundations | Confirm and expand | Narrow libraries deliberately exclude policy; Stackpress's distinct value is their operational coordination. |

## Retired Working Phrases

- `operations console`: reject; use operational policy/configuration until a real
  console exists.
- `one model, many interfaces`: retain only as shorthand candidate because events
  and config also drive interfaces.
- `product contract`: retain only for founder review; source supports narrower
  model/semantic contract language.
- `model-to-surfaces framework`: revise to a neutral research phrase such as
  model-to-capabilities-and-surfaces loop.

## Newly Discovered Concepts

- `NC-001`: Progressive specialization from queue to event to route to lifecycle.
- `NC-002`: Structure-versus-semantics separation in Idea metadata.
- `NC-003`: Event-driven generator discovery before transformation.
- `NC-004`: Generated client as executable runtime registry.
- `NC-005`: Host-owned policy above deliberately narrow libraries.
- `NC-006`: Inspectable intermediate artifacts before execution.
- `NC-007`: Edge adaptation preserving native resources and escape hatches.
- `NC-008`: Package-style reuse across schemas, plugins, pages, and generated code.
- `NC-009`: Granular exports as generated-code compatibility contracts.
- `NC-010`: Server capability authority with client hydration as enhancement.
- `NC-011`: Form/view metadata symmetry over generic UI components.
- `NC-012`: Skills as part of framework distribution and contribution design.

## Required Founder Review

The technical conclusions are source-backed. Public positioning still requires
decisions about the meaning of “content management framework,” whether Stackpress
should claim broader application-framework scope, and which neutral research
phrases should become official project language.
